# Feature Specification: 部屋・棚管理

**Feature Branch**: `005-storage-location-management`  
**Created**: 2026-04-27  
**Status**: Complete  
**Input**: User description: "登録されている棚・部屋を一覧で確認し、部屋・棚を追加・編集・削除できるようにする。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 部屋と棚の一覧確認 (Priority: P1)

利用者として、登録済みの部屋と棚を一覧で確認し、どこに何を保管できるかを素早く把握したい。

**Why this priority**: 一覧確認は追加・編集・削除の前提操作であり、管理画面として最小価値を成立させるために最優先。

**Independent Test**: 部屋・棚データが存在する状態で画面を開き、一覧表示に部屋名・棚名・説明が表示されること、更新後に再表示で反映されることを確認すれば単独で価値検証できる。

**Acceptance Scenarios**:

1. **Given** 部屋と棚が1件以上登録済みである, **When** 利用者が保管場所管理画面を表示する, **Then** 登録済みの部屋と棚が識別可能な形式で一覧表示される
2. **Given** 部屋または棚が未登録である, **When** 利用者が画面を表示する, **Then** 空状態メッセージと追加導線が表示される

---

### User Story 2 - 部屋と棚の追加・編集 (Priority: P2)

利用者として、部屋や棚をポップアップから追加・編集し、保管場所情報を最新に保ちたい。

**Why this priority**: 日常運用で最も発生頻度が高いメンテナンス操作であり、一覧確認に続く主要価値となるため。

**Independent Test**: ポップアップで必須項目を入力して保存し、一覧に反映されること、編集で内容変更が保存されることを確認すれば単独で機能検証できる。

**Acceptance Scenarios**:

1. **Given** 利用者が部屋の追加または編集を開始した, **When** 「部屋」「説明」を入力して保存する, **Then** 一覧に新規または更新内容が反映される
2. **Given** 利用者が棚の追加または編集を開始した, **When** 「部屋」を選択し「棚」「説明」を入力して保存する, **Then** 一覧に新規または更新内容が反映される
3. **Given** 既に同名の部屋または棚が存在する, **When** 利用者が同名で保存しようとする, **Then** 保存は拒否され重複不可の理由が表示される

---

### User Story 3 - 部屋と棚の削除 (Priority: P3)

利用者として、不要になった部屋や棚を削除し、管理情報を整理したい。

**Why this priority**: 運用上重要だが、一覧確認と追加・編集が先に成立していれば業務継続は可能なため優先度はP3。

**Independent Test**: アイテム未紐づけの部屋・棚を削除できること、紐づけ済みは削除拒否されることを確認すれば単独で検証できる。

**Acceptance Scenarios**:

1. **Given** アイテムが紐づいていない部屋または棚がある, **When** 利用者が削除を確定する, **Then** 対象が削除され一覧から消える
2. **Given** アイテムが紐づいている部屋または棚がある, **When** 利用者が削除を確定する, **Then** 削除は拒否され紐づき解消が必要である旨が表示される

### Edge Cases

- 保存直前に同名データが別操作で作成されていた場合は、重複エラーとして保存を失敗させる（ただし、判定対象はアクティブなレコードのみ）
- 部屋または棚の「説明」フィールドが空の場合は保存できず、入力必須メッセージを表示する
- 棚追加時に部屋が未選択の場合は保存できず、選択必須メッセージを表示する
- 以前削除（論理削除）した部屋または棚と同じ名称で新規作成しようとした場合、新規レコードとして作成を許可する
- 削除対象が既に他操作で削除済みの場合は、一覧再取得と整合した最新状態を表示する
- 通信失敗時は入力内容を保持したまま再試行導線を提示する

## Requirements *(mandatory)*

### Functional Requirements

#### Must

- **FR-001**: システムは、登録済みの部屋と棚の一覧を利用者が1画面で確認できるようにしなければならない
- **FR-001a**: システムは、部屋・棚の一覧をすべての認証済み利用者で共通表示し、利用者による編集・削除権限の差別化を行わないようにしなければならない
- **FR-002**: システムは、部屋の追加・編集時に「部屋」「説明」を入力できるポップアップを提供し、保存結果を一覧へ反映しなければならない
- **FR-002a**: システムは、部屋の「説明」フィールドを必須項目とし、未入力時は保存エラーとしなければならない
- **FR-002b**: システムは、部屋名の最大長を50文字とし、超過時は入力不可またはエラーメッセージを表示しなければならない
- **FR-002c**: システムは、部屋の説明フィールドの最大長を200文字とし、超過時は入力不可またはエラーメッセージを表示しなければならない
- **FR-003**: システムは、棚の追加・編集時に「部屋」選択と「棚」「説明」入力が可能なポップアップを提供し、保存結果を一覧へ反映しなければならない
- **FR-003a**: システムは、棚の「説明」フィールドを必須項目とし、未入力時は保存エラーとしなければならない
- **FR-003b**: システムは、棚名の最大長を50文字とし、超過時は入力不可またはエラーメッセージを表示しなければならない
- **FR-003c**: システムは、棚の説明フィールドの最大長を200文字とし、超過時は入力不可またはエラーメッセージを表示しなければならない
- **FR-004**: システムは、部屋名および棚名の重複登録を許可してはならない（ただし、アクティブな状態のレコードのみを重複判定対象とし、論理削除されたレコードは対象外とする）
- **FR-005**: システムは、アイテムが紐づく部屋または棚の削除を許可してはならない
- **FR-005a**: システムは、部屋または棚の削除を物理削除ではなく論理削除で実装し、削除後も監査・復旧に対応できるようにしなければならない
- **FR-005b**: システムは、論理削除済みの部屋または棚と同じ名称で新規作成を試みた場合、新規レコードとして作成を許可しなければならない
- **FR-006**: システムは、追加・編集・削除操作を保存先データへ反映し、再表示時にも同じ状態を再現しなければならない

#### Should

- **FR-007**: システムは、入力不備や業務制約違反時に、利用者が次に取るべき行動がわかるメッセージを表示することが望ましい
- **FR-008**: システムは、既存の同等UIがある場合は一貫性を保つため再利用を優先することが望ましい

#### Could

- **FR-009**: システムは、一覧画面に並び順や簡易絞り込みを追加できる拡張余地を持つことができる

### Key Entities *(include if fe、削除フラグを持ち、複数の棚を含む。論理削除時は削除フラグが立ち、一覧表示対象外となる
- **棚 (Shelf)**: 部屋に属する保管場所の下位単位。部屋参照、名称、説明、削除フラグを持つ。論理削除時は削除フラグが立ち、一覧表示対象外となる
- **部屋 (Room)**: 保管場所の上位単位。名称、説明を持ち、複数の棚を含む
- **棚 (Shelf)**: 部屋に属する保管場所の下位単位。部屋参照、名称、説明を持つ
- **アイテム紐づき (Item Assignment)**: アイテムが部屋・棚に関連付く状態。削除可否判定に利用される

## Success Criteria *(mandatory)*

### Measurable Outcomes
文字数超過による入力エラーが、リアルタイム検証または保存時に100%検出される
- **SC-005**: 
- **SC-001**: 利用者の90%以上が、初回操作で3分以内に目的の部屋または棚を一覧から特定できる
- **SC-002**: 有効な入力での部屋・棚の追加または編集操作の95%以上が、30秒以内に完了する
- **SC-003**: 重複名称登録と紐づきあり削除の不正操作が、受け入れ試験で100%拒否される
- **SC-004**: リリース後1か月以内に、保管場所情報の誤登録・誤削除に関する問い合わせ件数が現行比30%以上減少する

## Assumptions

- 部屋・棚情報は全利用者で共通であり、すべての認証済み利用者が編集・削除を実行できる（利用者ごとの権限制御は対象外）
- UIは既存デザイン資産に準拠し、操作導線と文言も既存方針を踏襲する
- 対象は複数利用者による共有管理を前提とし、利用者個別の削除ロック機構
- 対象は単一利用者の手動更新を前提とし、他利用者操作のリアルタイム反映は対象外とする
- 既存のデータ保存基盤が利用可能であり、保存後の再取得で最新状態を表示できる

## Clarifications

### Session 2026-04-27

- Q: 説明フィールドの必須性はどうするか？ → A: 必須。保存時に説明なし時はエラー
- Q: 削除時のデータ処理方式は？ → A: 論理削除。ただし削除済みと同じ名称の再登録は許可し、新規レコードとして作成される
- Q: 部屋・棚情報の共有範囲は？ → A: 全利用者で共通。全利用者が同じ部屋・棚を確認・編集・削除できる
- Q: 入力フィールドの最大文字数制限は？ → A: 部屋名・棚名 50 文字、説明 200 文字

## 用語

- **部屋**: 保管場所の大分類
- **棚**: 部屋配下の保管場所小分類
- **紐づき**: アイテムが部屋・棚に割り当てられている関係
- **重複登録**: 既存と同一名称で新規作成または更新しようとする操作

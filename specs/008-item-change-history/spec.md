# 機能仕様: アイテム変更履歴

**フィーチャーブランチ**: `008-item-change-history`
**作成日**: 2026-05-05
**ステータス**: ドラフト
**入力**: ユーザーによる説明: "登録されているアイテムの変更履歴を保存・確認する"

## 明確化

### セッション 2026-05-06

- Q: アイテム更新は成功したが変更履歴の記録に失敗した場合、どう処理するか？ → A: 同一トランザクション内で処理し、履歴記録失敗時はアイテム更新もロールバックする
- Q: 変更履歴を保存するタイミングはどうするか？ → A: アイテム作成・更新 API が呼ばれたときにサーバーサイドで自動記録する
- Q: 履歴カードに表示する変更内容の説明文フォーマットはどうするか？ → A: 変更後の値のみ記載する（例:「数量が5個に増加」「名称が"掃除機"に変更」）
- Q: 変更履歴データはどのように永続化するか？ → A: 既存 DB に専用テーブル（ItemHistory）として保存する
- Q: 論理削除されたアイテムの変更履歴はどう扱うか？ → A: 履歴データは保持し、削除済みアイテムの詳細非表示によって間接的にアクセス不可にする

## 目的（Why）

- 登録されているアイテムの変更履歴を保存する
- 登録されているアイテムの変更履歴を確認する

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - アイテム操作時に変更履歴を自動記録する (優先度: P1)

利用者として、アイテムを作成・更新したとき、その変更内容がシステムに自動的に記録されることを期待する。記録された履歴は後から確認でき、変更の経緯を把握できる。

**この優先度の理由**: 履歴の表示は記録がなければ意味をなさないため、記録機能が先行して成立しなければならない。

**独立テスト**: アイテムの作成・更新後に、対応する変更履歴レコードが正しく保存されていることを API または DB で確認することで独立してテストできる。

**受け入れシナリオ**:

1. **Given** アイテムが存在しない, **When** 利用者が新規アイテムを作成する, **Then** 「アイテムが作成された」旨の変更履歴が記録される
2. **Given** アイテムが存在する, **When** 利用者がアイテムの数量を更新する, **Then** 「数量が○個に変更された」旨の変更履歴が記録される
3. **Given** アイテムが存在する, **When** 利用者がアイテムの値段を更新する, **Then** 「値段が○円に変更された」旨の変更履歴が記録される
4. **Given** アイテムが存在する, **When** 利用者がアイテムの名称を更新する, **Then** 「名称が○○に変更された」旨の変更履歴が記録される
5. **Given** アイテムが存在する, **When** 利用者がアイテムのカテゴリを更新する, **Then** 「カテゴリが○○に変更された」旨の変更履歴が記録される
6. **Given** アイテムが存在する, **When** 利用者が複数項目（例: 名称と数量）を同時に更新する, **Then** 更新された項目の数だけ、それぞれ個別の変更履歴が記録される（同一日時で複数件）

---

### ユーザーストーリー 2 - アイテム詳細ページで変更履歴を確認する (優先度: P1)

利用者として、アイテム詳細ページを開いたとき、対象アイテムの直近の変更履歴を確認し、いつ・何が変わったかを把握したい。

**この優先度の理由**: アイテム変更履歴の確認は本機能の主要なユーザー価値であり、記録機能と並んで最優先で成立させる必要がある。

**独立テスト**: アイテム詳細ページを開き、変更履歴エリアに直近5件以内の履歴（変更内容・日時）が正しく表示されることを確認することで独立してテストできる。

**受け入れシナリオ**:

1. **Given** アイテムの変更履歴が存在する, **When** 利用者がアイテム詳細ページを開く, **Then** 直近5件以内の変更履歴が変更内容と変更日時（JST）とともに表示される
2. **Given** アイテムの変更履歴が5件を超えて存在する, **When** 利用者がアイテム詳細ページを開く, **Then** 最新5件のみが表示される
3. **Given** アイテムが作成されただけで更新履歴がない, **When** 利用者がアイテム詳細ページを開く, **Then** 「作成」の履歴1件が表示される
4. **Given** 同一日時に複数項目が更新された履歴が存在する, **When** 利用者がアイテム詳細ページを開く, **Then** 更新された項目ごとに1件ずつ履歴が表示される
5. **Given** アイテムの変更履歴が存在しない, **When** 利用者がアイテム詳細ページを開く, **Then** 履歴なしを示す適切なメッセージが表示される

---

### ユーザーストーリー 3 - 変更種別に応じた視覚的区別で履歴を把握する (優先度: P2)

利用者として、変更履歴の種別（新規作成・数量増加・数量減少・その他更新）が色によって視覚的に区別されており、内容を素早く識別したい。

**この優先度の理由**: 機能として必須ではないが、履歴の視認性を向上させるユーザー体験の改善であるため P2 とする。

**独立テスト**: アイテム詳細ページの変更履歴エリアで、各変更種別に応じた色（青系・グリーン系・中立系・黄系）が正しく適用されていることを目視または CSS クラス確認で独立してテストできる。

**受け入れシナリオ**:

1. **Given** アイテムの新規作成履歴が表示されている, **When** 利用者が履歴カードを確認する, **Then** 青系の色でカードが表示される
2. **Given** 数量が増加した履歴が表示されている, **When** 利用者が履歴カードを確認する, **Then** `recent-item.positive` クラスが適用されたスタイルでカードが表示される
3. **Given** 数量が減少した履歴が表示されている, **When** 利用者が履歴カードを確認する, **Then** `.recent-item.neutral` クラスが適用されたスタイルでカードが表示される
4. **Given** 名称・値段・カテゴリの更新履歴が表示されている, **When** 利用者が履歴カードを確認する, **Then** 黄系の色でカードが表示される

---

### エッジケース

- アイテムに変更履歴が1件も存在しない場合、履歴なしを示すメッセージを表示し、エラー扱いにしない
- 変更履歴の取得に失敗した場合は、アイテム詳細の他の情報表示を妨げずにエラー状態を表示し、再試行可能な導線を提供する
- 複数項目を同時更新した場合に記録される複数件の履歴は、件数カウントとして5件制限に含める
- アイテム詳細ページが表示された時点の履歴を表示する（ページ滞在中のリアルタイム更新は対象外）
- アイテム更新と変更履歴記録は同一トランザクションで処理し、いずれかが失敗した場合は両方をロールバックする（部分的な成功は許容しない）
- 論理削除されたアイテムの変更履歴データは DB に保持するが、削除済みアイテムの詳細ページが表示されないため利用者からはアクセス不可となる（追加の削除処理は不要）

## 要件 *(必須)*

### 機能要件

- **FR-001**: アイテムが新規作成されたとき、システムはサーバーサイドで自動的に「作成」種別の変更履歴を同一トランザクション内で記録すること
- **FR-002**: アイテムの数量が更新されたとき、システムはサーバーサイドで自動的に更新後の値と変化方向（増加/減少）を含む変更履歴を同一トランザクション内で記録すること
- **FR-003**: アイテムの値段が更新されたとき、システムはサーバーサイドで自動的に更新後の値を含む変更履歴を同一トランザクション内で記録すること
- **FR-004**: アイテムの名称が更新されたとき、システムはサーバーサイドで自動的に更新後の名称を含む変更履歴を同一トランザクション内で記録すること
- **FR-005**: アイテムのカテゴリが更新されたとき、システムはサーバーサイドで自動的に更新後のカテゴリ名を含む変更履歴を同一トランザクション内で記録すること
- **FR-006**: 同一操作で複数項目が更新された場合、システムは更新された項目の数だけ個別の変更履歴を同一日時で記録すること
- **FR-007**: アイテム詳細ページを表示したとき、システムは対象アイテムの変更履歴を最新順に最大5件取得して表示すること
- **FR-008**: 変更履歴の各項目には変更後の値を含む説明文（例:「数量が5個に増加」「名称が"掃除機"に変更」）と変更日時（JST）を表示すること。説明文には変更前の値は含めない
- **FR-009**: 変更履歴の表示には既存の `recent-activity-card` クラスを利用すること
- **FR-010**: 新規作成の履歴は青系の色スタイル（新規定義）を適用すること
- **FR-011**: 数量増加の履歴は `recent-item.positive` クラスを適用すること
- **FR-012**: 数量減少の履歴は `.recent-item.neutral` クラスを適用すること
- **FR-013**: 名称・値段・カテゴリ更新の履歴は黄系の色スタイル（新規定義）を適用すること
- **FR-014**: 変更履歴が存在しない場合は履歴なしを示すメッセージを表示すること

### 主要エンティティ

- **変更履歴 (ItemHistory)**: アイテム操作ごとに記録されるイベント。既存 DB の専用テーブルとして永続化する。対象アイテムID・変更種別・変更内容説明・変更日時を持つ
- **変更種別**: 作成 / 数量更新（増加） / 数量更新（減少） / 値段更新 / 名称更新 / カテゴリ更新 の列挙

## 成功基準 *(必須)*

### 測定可能な成果

- **SC-001**: アイテムを作成または更新したとき、変更履歴が100%の操作で記録される
- **SC-002**: 利用者はアイテム詳細ページを開くだけで、直近5件の変更履歴を追加操作なしに確認できる
- **SC-003**: 変更種別を視覚的に区別できるため、利用者は色を手がかりに変更内容を即座に識別できる
- **SC-004**: 複数項目の同時更新時も各項目の変更が漏れなく記録され、利用者が変更の全容を把握できる

## 前提条件

- アイテム詳細ページ（006-item-detail-page）の実装が完了していること
- アイテムの作成・更新 API が実装済みであること
- `recent-activity-card` クラスを含む既存 UI コンポーネントが利用可能であること
- `recent-item.positive` / `.recent-item.neutral` クラスが既存スタイルシートに定義済みであること
- 変更日時は UTC で保存し、表示時に JST（UTC+9）へ変換する
- 変更履歴は既存 DB の ItemHistory 専用テーブルに保存する
- v1 では変更履歴の削除・編集・ページネーションはスコープ外とする
- 変更を行ったユーザーの記録はスコープ外とする（単一ユーザー想定）

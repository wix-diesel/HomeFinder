# Research: アイテム保管場所表示

**日付**: 2026-05-29  
**対象機能**: 016-item-storage-location

## 目的

アイテム編集画面と詳細画面で扱う部屋・棚情報の保存/表示仕様を、既存アーキテクチャと整合した形で確定する。

## 調査と決定

### 1) 部屋・棚の任意入力と整合性
- Decision: 部屋・棚はどちらも未設定保存を許可する。ただし棚を設定する場合は部屋必須とする。
- Rationale: 既存データとの互換性を維持しつつ、部屋配下概念としての棚の整合性を守れるため。
- Alternatives considered: 部屋・棚とも必須にする案（既存データ移行負荷が高い）、棚単独設定を許可する案（整合性が崩れる）。

### 2) 削除済みマスター参照時の表示
- Decision: 削除済みの部屋/棚を参照するアイテムは「削除済み（元の名称）」として表示する。
- Rationale: 履歴的な意味を保持し、利用者が値消失と誤認しないため。
- Alternatives considered: 未設定表示に置換する案（情報欠落）、表示エラーにする案（閲覧阻害）。

### 3) 候補取得失敗時の編集挙動
- Decision: 部屋・棚編集のみ無効化し、その他項目の編集保存は継続可能にする。
- Rationale: 部分障害時でも主要な編集作業を止めない方が運用上の影響が小さいため。
- Alternatives considered: 画面全体保存不可（業務停止リスク）、古い候補キャッシュ継続（誤選択リスク）。

### 4) 用語統一
- Decision: 編集・詳細とも正規用語は「部屋」「棚」とし、「保存先」は廃止する。
- Rationale: 画面間の語彙差による誤解を防ぎ、受け入れテスト観点を統一できるため。
- Alternatives considered: 同義語併記（仕様と実装の対応が曖昧）。

### 5) バックエンド実装パターン
- Decision: API は既存 Item 更新/取得契約を拡張し、Application サービスは DotNext.Result<T> を返す既存方針を踏襲する。
- Rationale: 憲章の API-First とオニオンアーキテクチャ原則を満たし、既存呼び出しとの互換性を保てるため。
- Alternatives considered: フロントのみ状態保持（永続化要件を満たせない）、別 API を乱立（契約複雑化）。

## 実装前チェック項目

1. アイテム更新 API の契約で roomId/shelfId を nullable として扱うこと。
2. 削除済み名称表示のため、参照先名称解決のフォールバック仕様を API 契約に明記すること。
3. 候補取得失敗時の UI 状態（入力無効・他項目保存可）を UI 契約に明記すること。
4. 既存アイテム（roomId/shelfId null）表示の回帰テストケースを tasks で必須化すること。

## SC-003 ユーザー調査の実施計画

### 目的

- 編集画面サマリーに「部屋」「棚」を分離表示した変更が、情報理解を改善するかを評価する。

### 対象とサンプル

- 対象: 日常的に在庫編集を行う利用者
- サンプル数: 5〜10 名

### 手順

1. 既存版（保存先表示）と新UI（部屋/棚分離表示）を同一タスクで比較
2. タスク: 指定アイテムの保管場所を確認し、正しい棚へ修正して保存
3. タスク完了後に5段階アンケートを実施

### 評価項目

- 主観評価: 「保管場所の把握しやすさ」5段階
- 客観評価: タスク完了時間、誤操作回数、再確認回数

### 合格基準

- 主観評価平均が 4.0 以上
- 既存版比で誤操作回数が改善、または同等

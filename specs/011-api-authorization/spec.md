# 機能仕様: API認可設定

**フィーチャーブランチ**: `011-api-authorization`  
**作成日**: 2026-05-08  
**ステータス**: ドラフト  
**入力**: ユーザーによる説明: "アプリに認可設定を行う"

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - 適切な権限を持つユーザーによるAPI操作 (優先度: P1)

ログイン済みユーザーが自身のロール（Items.Read / Items.Create / Items.Delete / User）に対応した操作を行うと、APIが正常にレスポンスを返す。フロントエンドはAPIリクエスト時にトークンをヘッダーに付与して送信する。

**この優先度の理由**: 認可の基本動作であり、すべてのAPI操作の前提となる。これが機能しない場合、アプリ全体が使用不能になる。

**独立テスト**: Items.Read ロールを持つユーザーがアイテム一覧ページを開き、アイテムが正常に表示されることで独立してテストできる。

**受け入れシナリオ**:

1. **Given** Items.Read ロールを持つログイン済みユーザーが、**When** アイテム一覧・詳細ページを表示する、**Then** APIが正常にデータを返し、画面にアイテム情報が表示される
2. **Given** Items.Create ロールを持つログイン済みユーザーが、**When** 新規アイテムを登録する、**Then** APIがアイテムを正常に作成し、登録完了メッセージが表示される
3. **Given** Items.Create ロールを持つログイン済みユーザーが、**When** 既存アイテムを更新する、**Then** APIがアイテムを正常に更新し、更新完了メッセージが表示される
4. **Given** Items.Delete ロールを持つログイン済みユーザーが、**When** アイテムを削除する、**Then** APIがアイテムを正常に削除し、削除完了メッセージが表示される
5. **Given** User ロールを持つログイン済みユーザーが、**When** アイテム以外の操作（設定変更など）を行う、**Then** APIが正常にレスポンスを返す

---

### ユーザーストーリー 2 - 権限不足のユーザーがAPI操作を試みる (優先度: P1)

ログイン済みではあるが、操作に必要なロールを持っていないユーザーがAPIを呼び出そうとすると、フロントエンドにアクセス権がない旨のトースト通知が表示される。

**この優先度の理由**: 不正アクセスを防止し、ユーザーに適切なフィードバックを提供するために不可欠。

**独立テスト**: Items.Read のみのロールを持つユーザーがアイテム削除操作を試みた際にトーストが表示されることで独立してテストできる。

**受け入れシナリオ**:

1. **Given** Items.Read ロールのみを持つユーザーが、**When** アイテム作成・更新操作を実行する、**Then** APIが 403 Forbidden を返し、フロントエンドに「アクセス権がありません」トーストが表示される
2. **Given** Items.Read ロールのみを持つユーザーが、**When** アイテム削除操作を実行する、**Then** APIが 403 Forbidden を返し、フロントエンドに「アクセス権がありません」トーストが表示される
3. **Given** Items.Create ロールのみを持つユーザーが、**When** アイテム削除操作を実行する、**Then** APIが 403 Forbidden を返し、フロントエンドに「アクセス権がありません」トーストが表示される
4. **Given** いずれかのロールを持つユーザーが、**When** トークンなしでAPIを呼び出す、**Then** APIが 401 Unauthorized を返す

---

### エッジケース

- トークンの有効期限が切れた状態でAPIを呼び出した場合はどうなるか？（401 を返し、再ログインを促す）
- ロールが割り当てられていないユーザーがAPIを呼び出した場合はどうなるか？（403 を返し、トーストを表示する）
- フロントエンドでトークン取得に失敗した場合はどうなるか？（エラートーストを表示し、APIリクエストは送信しない）

## 要件 *(必須)*

### 機能要件

- **FR-001**: システムはバックエンドのアイテム取得API（一覧・詳細）に対して「Items.Read」ロールを要求すること
- **FR-002**: システムはバックエンドのアイテム作成・更新APIに対して「Items.Create」ロールを要求すること
- **FR-003**: システムはバックエンドのアイテム削除APIに対して「Items.Delete」ロールを要求すること
- **FR-004**: システムはアイテム系以外のAPI（カテゴリ管理・収納場所管理・設定操作など）に対して「User」ロールを要求すること
- **FR-005**: フロントエンドはすべてのAPIリクエスト時に認証トークンをリクエストヘッダーに付与して送信すること
- **FR-006**: フロントエンドはAPIから 403 Forbidden レスポンスを受け取った場合、「アクセス権がありません」というトースト通知を画面に表示し、3〜5秒後に自動で消えること
- **FR-007**: フロントエンドはAPIから 401 Unauthorized レスポンスを受け取った場合、MSALのサイレントトークン更新を試みること。更新に成功した場合は元のリクエストを再送し、失敗した場合はログインページへリダイレクトすること
- **FR-008**: 複数のロールを持つユーザーに対しては、OR条件でアクセス制御を評価すること（いずれか1つのロールが一致すればアクセス許可）

### 主要エンティティ

- **ロール（Role）**: ユーザーに割り当てられた権限の種類。Items.Read / Items.Create / Items.Delete / User の4種類
- **トークン（Token）**: Azure Entra から発行される認証・認可情報を含む JWT トークン。APIリクエスト時にヘッダーへ付与する
- **API操作**: アイテムの取得・作成・更新・削除、およびその他の設定操作

## 成功基準 *(必須)*

### 測定可能な成果

- **SC-001**: Items.Read ロールを持つユーザーが、アイテムの作成・更新・削除を試みた際に、必ず「アクセス権がありません」トーストが表示される（発生率 100%）
- **SC-002**: 適切なロールを持つユーザーが各API操作を行う際、認可エラーが発生しない（エラー率 0%）
- **SC-003**: 権限不足時のトースト通知が API レスポンス受信から 1 秒以内に表示され、3〜5秒後に自動で消える
- **SC-004**: すべてのAPIエンドポイントに対して認可チェックが適用されており、未保護のエンドポイントが存在しない

## 明確化

### セッション 2026-05-08

- Q: 複数ロールを持つユーザーのアクセス制御はOR条件かAND条件か → A: OR条件（各ロールは独立して評価し、いずれか1つが一致すればアクセス許可）
- Q: 401 Unauthorized 受信時のフロントエンド動作 → A: サイレントトークン更新を試み、失敗した場合のみログインページへリダイレクト
- Q: 「User」ロールが対象とするAPIの範囲 → A: カテゴリ管理・収納場所管理・設定操作など、アイテム系以外のすべてのAPI
- Q: 「アクセス権がありません」トーストの表示後の動作 → A: 数秒後（3〜5秒）に自動で消える
- Q: バックエンドAPIの認可チェックの実施レイヤー → A: 各APIコントローラー/エンドポイントの属性で宣言的に設定

## 前提条件

- Azure Entra を使用したログイン認証（010-azure-entra-login）が実装済みであること
- ユーザーのロールは Azure Entra（Microsoft Entra ID）上のアプリロールとして事前に設定されていること
- ロールの管理（付与・変更）はスコープ外とし、Azure ポータル上での手動設定を前提とする
- フロントエンドは MSAL（Microsoft Authentication Library）などを通じてトークンを取得できること
- トークンのロールに応じた UI の表示切り替えはスコープ外とする
- バックエンドの認可チェックは各APIエンドポイントの属性で宣言的に設定する（ミドルウェア一括処理やサービス層での検証は対象外）

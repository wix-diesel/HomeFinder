# Feature Specification: アイテム画面デザイン再構成

**Feature Branch**: `002-before-specify-hook`  
**Created**: 2026-04-24  
**Status**: Draft  
**Input**: User description: "UIのデザインを洗練させ、アイテム一覧画面とアイテム作成画面をデザイン参照に基づいて再作成し、再利用しやすいコンポーネント設計と日本語UIを実現する"

## 目的（Why）

- UIを洗練し、利用者が迷わず操作できる一覧・登録体験を提供する。
- 将来のページ追加に対応できるよう、画面間で再利用可能なUI部品を定義し、開発と保守の効率を高める。

## Clarifications

### Session 2026-04-24

- Q: 未定義項目（現状 Item にない入力欄）を「登録」時にどう扱うか → A: 画面には表示・入力可能だが、登録API送信には含めない（UIのみ保持）
- Q: 登録成功後の遷移をどうするか → A: 一覧画面へ遷移し、成功トーストを表示する
- Q: 空状態・エラー状態・処理中状態の共通設計をどこまで必須化するか → A: 一覧画面・登録画面の両方で、空状態/検証エラー/送信中/失敗の4状態を共通UI部品として必須化する
- Q: レスポンシブ時の一覧表示レイアウトをどう定義するか → A: モバイルはカード、デスクトップはカードまたはテーブル切替を許可する
- Q: 日本語化の適用範囲をどこまで必須にするか → A: 画面ラベル、ボタン、エラーメッセージ、ヘルプテキスト、空状態文言の全てを日本語必須にする

## User Scenarios & Testing *(mandatory)*

### User Story 1 - アイテム一覧を素早く把握する (Priority: P1)

在庫担当者として、アイテム一覧画面で在庫状況・カテゴリ・価格などの主要情報をカード形式で素早く把握したい。これにより、次の業務判断（補充、確認、詳細確認）を短時間で行える。

**Why this priority**: 一覧確認は日常業務で最も頻度が高く、ここが使いにくいと全体効率が大きく低下するため。

**Independent Test**: 一覧画面単体で、検索・カテゴリ絞り込み・在庫ステータス視認が成立し、ユーザーが目的のアイテム候補を特定できることを確認できる。

**Acceptance Scenarios**:

1. **Given** アイテム一覧画面を開いている, **When** ユーザーがキーワード検索する, **Then** 条件に合うカードのみが即時に表示される。
2. **Given** 複数カテゴリのアイテムが存在する, **When** ユーザーがカテゴリチップを選択する, **Then** 選択カテゴリに一致するカードだけが表示される。
3. **Given** 一覧カードに在庫情報がある, **When** 画面を閲覧する, **Then** 在庫ステータス（在庫あり/残り少ない/在庫切れ）が視覚的に区別できる。

---

### User Story 2 - アイテムを迷わず登録する (Priority: P1)

在庫担当者として、登録画面で必要項目を日本語UIで入力し、新規アイテムを登録したい。これにより、入力漏れや誤操作を減らして登録作業を完了できる。

**Why this priority**: 一覧と並ぶ主要業務であり、登録体験が不明瞭だと運用データの品質が低下するため。

**Independent Test**: 登録画面単体で、入力項目の理解、必須項目入力、送信操作、エラー表示が完結することを確認できる。

**Acceptance Scenarios**:

1. **Given** ユーザーが登録画面を開いている, **When** 必須項目を入力して登録を実行する, **Then** 登録完了状態が明示される。
2. **Given** ユーザーが登録を成功させた, **When** 登録処理が完了する, **Then** 一覧画面へ遷移し成功トーストが表示される。
3. **Given** 必須入力が不足している, **When** ユーザーが登録を実行する, **Then** 不足項目が日本語メッセージで案内され再入力できる。
4. **Given** 現在のItem定義にない入力欄が画面に存在する, **When** ユーザーが画面を利用する, **Then** 該当欄もデザインどおり表示され、入力対象として扱える。

---

### User Story 3 - 一貫した画面操作で学習コストを下げる (Priority: P2)

利用者として、一覧画面と登録画面で同じ見た目・同じ操作パターンのUI部品を使いたい。これにより、新しい画面が増えても迷わず操作できる。

**Why this priority**: 今後の画面拡張時にUXの一貫性を保つことで、教育コストと操作ミスを抑制できるため。

**Independent Test**: 一覧/登録の2画面で共通部品（ヘッダー、入力部品、アクションボタン、カード要素）が同一ルールで適用されることを確認できる。

**Acceptance Scenarios**:

1. **Given** 一覧画面と登録画面を比較する, **When** 共通UI領域を確認する, **Then** タイポグラフィ、余白、ボタン状態、ラベル表現が統一されている。
2. **Given** 将来ページ用のUI部品一覧が存在する, **When** 開発者が新ページ設計を行う, **Then** 既存部品を再利用できる判断基準が明確である。

---

### Edge Cases

- 検索条件に一致するアイテムが0件の場合、空状態メッセージと次アクション（検索条件の解除）を表示する。
- カテゴリ絞り込みと検索語を同時に指定し結果が0件になった場合、現在の適用条件を明示する。
- 登録画面で数値項目に不正値（0未満、文字列）が入力された場合、該当項目単位で日本語エラーを表示する。
- 登録画面に存在するが現時点でドメイン未定義の項目は、入力欄として表示しつつ、保存対象外の場合はユーザーが誤解しない補足を表示する。
- 通信遅延や登録失敗時は入力内容を保持し、再試行できる状態を維持する。
- 送信中状態では重複送信を防ぐため主要アクションを無効化し、進行中であることを明示する。
- 一覧取得失敗時は失敗状態コンポーネントを表示し、再読み込み操作を提供する。

## Requirements *(mandatory)*

### 要求（Must / Should / Could）

#### Must

- 一覧画面は `design/items_list.html` の情報設計を基準に再構成し、カード単位で主要情報を視認できなければならない。
- 登録画面は `design/item_register.html` の情報設計を基準に再構成し、日本語UIで操作できなければならない。
- 登録画面には、現状Item定義にない項目が含まれていても、参照デザインどおり表示しなければならない。
- 一覧画面と登録画面で共通化可能なUI部品（ヘッダー、ボタン、フォーム入力、カード構造、状態表示）を再利用前提で定義しなければならない。
- 画面ラベル、ボタン、エラーメッセージ、ヘルプテキスト、空状態文言はすべて日本語で提供しなければならない。

#### Should

- 一覧と登録の導線は、初見ユーザーが「一覧表示から登録開始」までを2クリック以内で到達できるよう、用語・配置・操作フィードバックを統一すべきである。
- エラー表示、空状態表示、処理中表示は、文言と視覚スタイルを共通ルール化すべきである。

#### Could

- 将来ページ追加時に使えるUIパターン一覧（例: 検索領域、フィルタチップ群、アクションバー）を整備してよい。

### Functional Requirements

- **FR-001**: システムは、検索・カテゴリ絞り込み・ステータス視認可能なアイテムカードを含む、参照一覧デザインに基づいて再構成したアイテム一覧画面を提供しなければならない。
- **FR-002**: システムは、参照登録デザインに基づき、入力セクション構成と主要アクション配置を備えたアイテム登録画面を提供しなければならない。
- **FR-003**: システムは、Item ドメインモデルに現在定義されていない項目であっても、参照デザインに存在する登録入力項目を画面に表示しなければならない。
- **FR-003a**: システムは、UI専用項目に入力された値をクライアント側で保持し、登録APIのリクエストpayloadには含めてはならない。
- **FR-004**: システムは、同一のUI部品を将来ページへ一貫適用できるよう、再利用可能なUIコンポーネントと利用ルールを定義しなければならない。
- **FR-005**: システムは、空状態・検証エラー・登録失敗時に、失敗理由と少なくとも1つの復旧操作（例: 再試行、入力修正）を含む日本語メッセージを表示しなければならない。
- **FR-006**: ユーザーは、英語表記に依存せずに、一覧画面から登録開始操作までを2クリック以内で完了できなければならない。
- **FR-007**: システムは、登録成功後にユーザーをアイテム一覧画面へ遷移させ、成功トーストメッセージを表示しなければならない。
- **FR-008**: システムは、4状態（空状態・検証エラー・送信中・失敗）用の共通UIコンポーネントを実装し、一覧画面と登録画面の両方へ一貫適用しなければならない。
- **FR-009**: システムは、モバイルでは一覧をカードレイアウトで表示し、デスクトップではカード/テーブル切替を可能にしなければならない。
- **FR-010**: システムは、一覧画面と登録画面のラベル・ボタン・検証エラー・ヘルプテキスト・空状態メッセージを日本語で提供しなければならない。

### Key Entities *(include if feature involves data)*

- **一覧カード表示情報**: アイテム名、カテゴリ、在庫数、在庫ステータス、価格など、一覧カードで視認する情報単位。
- **登録フォーム入力セット**: アイテム登録時に入力する項目群。現行ドメイン定義済み項目と、参照デザイン上の追加項目を含む。
- **UIコンポーネント定義**: ボタン、入力欄、ラベル、カード、ステータス表示など再利用可能な部品と適用ルール。
- **画面状態メッセージ**: 空状態、検証エラー、失敗通知、処理中通知などユーザー案内の状態表現。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: アプリ起動から一覧確認完了までが 2 分以内である。
- **SC-002**: 有効な登録リクエストの成功率が 95% 以上である。
- **SC-003**: 空状態・入力エラー・登録失敗におけるエッジケース処理の完全性が 100% である。
- **SC-004**: UI-only 項目非送信・レスポンシブ表示切替に関するエッジケース処理の完全性が 100% である。
- **SC-005**: 一覧画面と登録画面のラベル、ボタン、エラーメッセージ、ヘルプテキスト、空状態文言の100%が日本語で表示される。
- **SC-006**: 新規ページ追加時に、定義済み共通UI部品を使って画面要素の70%以上を再利用できる（再利用率 = 再利用した対象UI要素数 / 対象UI要素総数。対象UI要素はヘッダー、ボタン、入力欄、状態表示、カード、表示切替とする）。
- **SC-007**: 空状態・入力エラー・登録失敗時の再操作完了率が95%以上になる。
- **SC-008**: 一覧画面・登録画面で、4状態（空状態/検証エラー/送信中/失敗）の共通UI部品適用率が100%である。
- **SC-009**: モバイル表示では一覧がカード形式で表示され、デスクトップ表示ではカード/テーブル切替操作が100%利用可能である。

## Assumptions

- 対象ユーザーは在庫管理担当者であり、PCブラウザでの利用を主とする。
- この仕様で扱う範囲は、一覧画面と登録画面のUI再設計および再利用設計までとし、認証・同期・共有は含まない。
- 写真撮影登録、バーコードスキャン、ドラフト登録は今回の実装対象外とする。
- 現在Itemに未定義の登録項目は、UI上で提示・入力可能としつつ、登録APIの送信対象には含めない。
- 参照デザインの英語テキストは、意味を維持した日本語表現に置き換える。

## 用語

- **一覧画面**: アイテムをカードまたは同等の視認単位で表示し、検索・絞り込みを行う画面。
- **登録画面**: 新規アイテム情報を入力して登録操作を行う画面。
- **共通UI部品**: 複数画面で再利用される見た目と挙動を持つ画面構成要素。
- **画面状態**: 空状態、入力エラー、処理中、失敗、完了など、ユーザーに提示されるUIの状態。

# 機能仕様: Azure Entra ログイン認証

**フィーチャーブランチ**: `010-azure-entra-login`  
**作成日**: 2026-05-07  
**ステータス**: ドラフト  
**入力**: ユーザーによる説明: "アプリにログインしたユーザーのみ、ページを閲覧できるようにする"

## ユーザーシナリオとテスト *(必須)*

### ユーザーストーリー 1 - 未認証ユーザーのログインページ遷移 (優先度: P1)

未ログインのユーザーがアプリのいずれかのページにアクセスしようとすると、自動的にログインページへリダイレクトされる。ログインページには「Sign in with Microsoft」ボタンが表示されており、押下するとMicrosoftのログインページに遷移する。

**この優先度の理由**: アプリ全体のアクセス制御の基盤となる機能であり、これなしでは他のすべての機能が未認証ユーザーに公開されてしまう。

**独立テスト**: ログアウト状態でアプリのURLにアクセスし、ログインページへリダイレクトされることと、Microsoftログインボタンが表示されることで独立してテストできる。

**受け入れシナリオ**:

1. **Given** 未ログインのユーザーが、**When** アプリの任意のページURL（ログインページ以外）にアクセスする、**Then** ログインページに自動リダイレクトされる
2. **Given** ログインページを開いたユーザーが、**When** ページを表示する、**Then** 「Sign in with Microsoft」ボタンが表示される
3. **Given** ログインページを開いたユーザーが、**When** 「Sign in with Microsoft」ボタンを押下する、**Then** Azure EntraのMicrosoftログイン画面に遷移する

---

### ユーザーストーリー 2 - Microsoftアカウントでのログインとリダイレクト (優先度: P1)

ユーザーがMicrosoftアカウントで認証に成功すると、ログイン前にアクセスしようとしていたページ（元のURL）に自動的にリダイレクトされる。

**この優先度の理由**: ログイン後のリダイレクト先が元のページであることはユーザー体験上不可欠であり、P1と同等の優先度を持つ。

**独立テスト**: 特定のページURL（例: アイテム詳細ページ）へアクセス → ログインページにリダイレクト → Microsoftアカウントで認証 → 元のURLに戻ることでテストできる。

**受け入れシナリオ**:

1. **Given** 特定のページURLにアクセスしようとしてログインページにリダイレクトされたユーザーが、**When** Microsoftアカウントで認証に成功する、**Then** アクセスしようとしていた元のページに遷移する
2. **Given** ログインページに直接アクセスしたユーザーが、**When** Microsoftアカウントで認証に成功する、**Then** アプリのデフォルトページ（トップまたはホーム画面）に遷移する
3. **Given** 認証済みのユーザーが、**When** ログインページに直接アクセスする、**Then** ログインページは表示されず、デフォルトページにリダイレクトされる

---

### ユーザーストーリー 3 - ログインUIデザインの再現 (優先度: P2)

ログインページのUIは `/design/login.html` で定義されたデザインに準拠する。ブランドカラー、レイアウト、ボタンデザイン（Microsoft四色ロゴを含む）、サポート情報メッセージを再現する。

**この優先度の理由**: UXとブランド一貫性のために重要だが、認証機能そのものよりは優先度が低い。

**独立テスト**: デザインファイルとの見た目の比較、および各UIコンポーネント（ヘッダー、ヒーロー画像エリア、ボタン、サポート情報、フッター）の存在確認でテストできる。

**受け入れシナリオ**:

1. **Given** ログインページを開いたユーザーが、**When** ページを表示する、**Then** アプリロゴ（Home Finder）がヘッダーに表示される
2. **Given** ログインページを開いたユーザーが、**When** ページを表示する、**Then** "Internal Access Only" の見出しと説明文が表示される
3. **Given** ログインページを開いたユーザーが、**When** ページを表示する、**Then** Microsoft四色ロゴを含む「Sign in with Microsoft」ボタンが表示される
4. **Given** ログインページを開いたユーザーが、**When** ページを表示する、**Then** ITサポートデスクへの連絡を促すサポート情報が表示される

---

### ユーザーストーリー 4 - ログアウト (優先度: P1)

ログイン済みのユーザーがアプリからログアウトできる。ログアウト時にはセッションを破棄し、Azure Entraへサインアウトを通知する。ログアウト後はログインページに遷移する。

**この優先度の理由**: ログアウト機能はログイン機能とセットで必要であり、特に共有端末でのセキュリティ確保に不可欠。

**独立テスト**: ログイン済み状態でログアウト操作を行い、セッションが破棄されてログインページに遷移し、再度保護されたページへのアクセスがリダイレクトされることで独立してテストできる。

**受け入れシナリオ**:

1. **Given** 認証済みのユーザーが、**When** ログアウト操作を行う、**Then** セッションが破棄され、Azure Entraへサインアウトが通知される
2. **Given** ログアウトが完了した後、**When** ログアウト処理が終わる、**Then** ログインページに遷移する
3. **Given** ログアウトしたユーザーが、**When** 保護されたページURLに直接アクセスする、**Then** ログインページにリダイレクトされる

---

### エッジケース

- 認証中にブラウザを閉じた場合、再度アクセス時にはログインページから再開される
- Microsoftログイン側でキャンセルした場合、ログインページに戻る（エラーメッセージは表示しない）
- セッション有効期限切れ後に操作した場合、ログインページにリダイレクトされる
- Azure Entraサービスが一時的に利用不可の場合、ユーザーにエラーを通知する（例: 「現在ログインできません。しばらくしてから再度お試しください」）
- Entraアカウントに登録されていないユーザーがログインを試みた場合、Microsoft側でエラーが表示される（アプリ側での処理は不要）
- ログアウト中にネットワークエラーが発生した場合、ローカルセッションは破棄し、ログインページに遷移する（Azure Entra側のサインアウトが不完全でも許容する）
- Azure Entraによる認証が失敗した場合（トークン検証エラー、テナント不一致等）、汎用エラーメッセージを表示してログインページに留まる。エラーの詳細原因は表示しない
- `returnUrl` に外部ドメインのURLが指定された場合（オープンリダイレクト攻撃の防止）、デフォルトページにリダイレクトする

## 要件 *(必須)*

### 機能要件

- **FR-001**: システムは、未認証ユーザーがログインページ以外のURLにアクセスした場合、`/login?returnUrl=<元のURL>` 形式でログインページに自動リダイレクトすること
- **FR-002**: システムは、ログインページに「Sign in with Microsoft」ボタンを表示し、押下するとAzure Entraの認証フローを開始すること
- **FR-003**: システムは、MSアカウント以外の認証方法（メール/パスワード、他のソーシャルログイン等）を提供しないこと
- **FR-004**: システムは、認証成功後にユーザーを元のアクセス先URLにリダイレクトすること
- **FR-005**: システムは、認証済みユーザーがログインページに直接アクセスした場合、デフォルトページにリダイレクトすること
- **FR-006**: システムは、ログインページのUIを `/design/login.html` に定義されたデザインに準拠して表示すること
- **FR-007**: システムは、Azure Entra（Microsoft Entra ID）を認証プロバイダーとして使用すること
- **FR-008**: システムは、ログアウト機能を提供すること。ログアウト時にはローカルセッションを破棄し、Azure Entraにサインアウトを通知すること
- **FR-009**: システムは、ログアウト完了後にログインページ（`/login`）へ遷移すること
- **FR-010**: システムは、認証失敗時（Azureトークン検証エラー・Azure Entra接続エラー・ネットワーク障害等）に汎用エラーメッセージ（例: 「ログインに失敗しました。再度お試しいただくか、ITサポートにご連絡ください」）を表示し、詳細なエラー原因は外部に公開しないこと
- **FR-011**: システムは、認証イベント（ログイン成功・失敗、ログアウト）をユーザーIDとタイムスタンプとともに記録すること

### 主要エンティティ *(機能にデータが関わる場合は含める)*

- **認証セッション**: ログインに成功したユーザーの認証状態。ユーザーIDとセッションの有効期限（24時間）を保持する
- **リダイレクトURL**: ログイン前にアクセスしようとしていたURL。ログインページのURLクエリパラメータ `returnUrl` として保持され、認証成功後の遷移先として使用する

## 成功基準 *(必須)*

### 測定可能な成果

- **SC-001**: 未認証ユーザーが保護されたページにアクセスした際、3秒以内にログインページへリダイレクトされる
- **SC-002**: ユーザーがMicrosoftアカウントで認証後、元のページへの遷移が2秒以内に完了する
- **SC-003**: ログインページのUIが `/design/login.html` のデザインと視覚的に一致している（すべての主要UIコンポーネントが存在する）
- **SC-004**: 認証済みユーザーは、ログインページを経由せず直接アプリを利用できる
- **SC-005**: Microsoftアカウント以外の認証手段がログインページに表示されない
- **SC-006**: 認証セッションは24時間有効であり、期限切れ後は自動的にログインページへリダイレクトされる
- **SC-007**: 認証イベント（ログイン成功・失敗、ログアウト）がユーザーIDとタイムスタンプとともに記録される

## 前提条件

- ユーザーは既にAzure Entra（Microsoft Entra ID）にアカウントが登録されている
- アプリケーションはAzure Entraに対してアプリ登録済みである（クライアントID等の設定は既に存在する、または別途設定される）
- ユーザー登録機能はスコープ外。新規ユーザー追加はAzure Entra管理者が行う
- 各APIエンドポイントの認可設定（ロールベースのアクセス制御等）は現フェーズのスコープ外
- アプリはブラウザベースのウェブアプリケーションである

## 明確化

### Session 2026-05-07

- Q: ログアウト機能はこのフィーチャーのスコープに含まれますか？ → A: スコープに含める（セッション破棄 + Azure Entraサインアウト）
- Q: セッションの有効期限はどのくらいにしますか？ → A: 24時間
- Q: `returnUrl` に外部ドメインのURLが指定された場合（オープンリダイレクト攻撃の防止）、デフォルトページにリダイレクトする → A: URLクエリパラメータ（`/login?returnUrl=/path`）
- Q: 認証失敗時（例: Azureトークンの検証エラー）にユーザーに表示するエラーメッセージはどうしますか？ → A: 汎用エラーメッセージを表示してログインページに留まる
- Q: ログインイベント（成功・失敗）の記録（ログ・監査証跡）はこのフィーチャーのスコープに含めますか？ → A: 最低限の認証イベントログをスコープに含める（成功・失敗のユーザーIDとタイムスタンプ）

## 用語

| 用語 | 説明 |
|------|------|
| Azure Entra / Microsoft Entra ID | Microsoftが提供するクラウドベースのIDおよびアクセス管理サービス（旧称: Azure Active Directory） |
| 認証 (Authentication) | ユーザーが本人であることを確認するプロセス |
| 認可 (Authorization) | 認証済みユーザーが特定のリソースにアクセスする権限を持つかを確認するプロセス（本機能のスコープ外） |
| リダイレクト | ユーザーを別のページやURLに自動的に転送すること |
| セッション | ユーザーのログイン状態を保持する仕組み |

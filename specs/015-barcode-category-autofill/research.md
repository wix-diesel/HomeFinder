# research.md

## 目的

Phase 0 の目的は、外部（楽天）API 仕様とカテゴリ自動登録の不確実性を解消し、実装方針（正規化・重複判定・リトライ方針）を決定することです。

---

## 調査項目と決定（案）

### 1) 楽天APIから取得するカテゴリ情報の扱い
- Decision: 楽天APIから取得できる「カテゴリ名（genre）」を一次情報とし、システム内の `Category.Name` にマッピングする。外部のカテゴリID（存在する場合）は `Category.ExternalId` に保存する。
- Rationale: 外部の分類は最小限の参照情報（表示名＋外部ID）で十分であり、内部ドメインの主キーはアプリ内で管理する方が柔軟性が高い。
- Alternatives considered: 外部IDを主キーとして連携する案（外部変更に脆弱）／外部カテゴリを参照のみで保持する案（検索・集計が困難）

### 2) カテゴリ名の正規化ルール
- Decision: 自動登録・重複判定には以下の正規化を適用し、正規化済み文字列を `Category.NormalizedName` に保存する。
  - 全角→半角変換（英数字・記号）
  - 前後の空白削除、連続スペースは単一スペースに正規化
  - 英字は小文字化
  - 全角カタカナ→ひらがな変換は行わない（現状の DB を踏襲し、必要なら後続で方針変更）
- Rationale: 単純な正規化で多くの表記差を吸収し、過剰な近似マッチによる誤登録を防ぐ。
- Alternatives considered: 文字列類似度（Levenshtein）によるマッチ — より柔軟だが誤判定リスクと運用コストが高い。

### 3) 重複判定アルゴリズム
- Decision: 重複判定は段階的に行う。
  1. 正規化後の完全一致を最優先で採用
  2. 完全一致がない場合は、閾値付きの近似一致（例: 正規化後の類似度 >= 0.9）を検討候補として管理者承認フローまたは UI 上で候補提示する（初期は自動採用しない）
- Rationale: 自動採用の誤登録を最小化しつつ、類似エントリ発見の利便性は維持するため。
- Alternatives considered: すべて手動判定にする（運用負荷が高い）／すべて自動採用する（誤登録リスク大）

### 4) 自動登録時の所有者・メタ情報
- Decision: 自動登録されるカテゴリの `CreatedBy` はシステムアカウント（例: `system:barcode-import`）とし、`Source` に `rakuten` を設定する。
- Rationale: 自動生成エントリはユーザー操作ではなく外部同期によるため、作成者を明確にし検索/監査を容易にする。
- Alternatives considered: 操作ユーザーを割り当てる（フロントエンドでの有効ユーザーも検討可能だが処理が複雑になる）

### 5) 楽天APIのレート制限とリトライ方針
- Decision: 楽天API のレート超過(HTTP 429) 時は exponential backoff を採用し、フロントエンド側では最大 2 回まで再試行する。バックエンドは 1 回のみ再試行し、最終的に失敗した場合は 503 を返してフロントにエラーメッセージを表示する。
- Rationale: ユーザーの待ち時間を長くしすぎない一方で、一時的なレート超過を取りこぼさないための折衷策。
- Alternatives considered: フロントで無制限再試行（ユーザー体験悪化）／再試行しない（回復可能な一時失敗を見逃す）

### 6) 同時登録・競合への対処
- Decision: 自動登録は DB の UNIQUE 制約（`NormalizedName`）で保護し、登録時に `409 Conflict` を検出した場合は既存レコードを再取得してマージ/利用する（楽観的リトライ）。
- Rationale: 同時性の問題は DB 制約で防ぎつつ、エラー時は既存レコードを利用することで冗長な重複登録を回避する。
- Alternatives considered: 排他ロック（性能低下）／事前チェックのみ（TOCTOU の問題が残る）

## 検証タスク（実行アクション）

1. 楽天APIのサンプルレスポンスを取得し、カテゴリ情報のフィールド（表示名、階層情報、外部ID）を抜粋する。
2. 既存 DB から現行カテゴリ名のサンプル 500 件を取得し、提案した正規化ルールを適用して衝突率（何 % が正規化後に一致するか）を評価する。
3. 正規化ルールに基づく重複検出の false-positive/false-negative 比率を小規模データで評価する（Levenshtein 閾値のチューニング）。
4. 楽天API の 429/5xx 発生状況を想定したローカル試験を行い、バックエンドの再試行ロジック（1 回）とフロントの UI 表示の確認を行う。
5. 同時登録シナリオ（複数プロセスが同一カテゴリを同時に登録）を DB でシミュレートし、`409` 検出→既存取得パスを検証する。

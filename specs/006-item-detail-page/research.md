# Research: アイテム詳細ページ操作

## Decision 1: 詳細ページの表示データ取得は既存アイテム詳細 API を利用する
- Decision: 詳細表示は既存の単一アイテム取得 API を利用し、新規の履歴取得 API は追加しない。
- Rationale: 本仕様は「詳細確認」「編集遷移」「削除」に限定され、履歴表示はスコープ外であるため、既存 API の再利用が最小変更で要件を満たす。
- Alternatives considered:
  - 新規の詳細専用集約 API を追加する: 表示最適化余地はあるが、スコープ過剰。
  - フロントエンドで一覧データを流用して詳細表示する: 最新性と整合性の担保が難しい。

## Decision 2: 削除は論理削除として実装する
- Decision: 削除確認で確定された場合は論理削除し、通常の一覧・詳細表示対象から除外する。
- Rationale: Clarifications で確定済み。監査性・復旧可能性を確保しつつ、UI 要件（削除後に一覧へ遷移）と整合する。
- Alternatives considered:
  - 物理削除: 復旧が困難。
  - 論理削除後に遅延物理削除: 運用要件未定で本機能には過剰。

## Decision 3: 削除成功後の遷移先はアイテム一覧に固定する
- Decision: 削除成功時は必ずアイテム一覧ページへ遷移する。
- Rationale: 削除済みリソースの詳細画面に留まらず、次アクションを明確化できる。
- Alternatives considered:
  - 詳細画面に留まり削除済み状態表示: 要件複雑化。
  - ブラウザ戻る依存: 遷移元により挙動が不安定。

## Decision 4: 競合（対象消失）時は失敗メッセージ後に一覧へ遷移する
- Decision: 編集保存・削除実行時に対象が存在しない場合は失敗扱いにし、メッセージ表示後に一覧へ遷移する。
- Rationale: 利用者に現状を明示しつつ、存在しないリソースへの操作ループを避ける。
- Alternatives considered:
  - 成功扱いで遷移: 操作結果の意味が曖昧になる。
  - 詳細画面に残留して再試行促進: 無効な再試行を誘発する。

## Decision 5: 認可は既存アイテム編集権限モデルを再利用する
- Decision: 詳細表示・編集・削除は既存のアイテム編集権限モデルを適用し、新規権限ロジックは導入しない。
- Rationale: 既存仕様との一貫性を保ち、変更影響を最小化できる。
- Alternatives considered:
  - 全員編集可: 将来の統制要件と衝突しやすい。
  - 管理者限定へ変更: 本機能要求外で仕様差分が大きい。

## Decision 6: UI は design/items_detail.html を基準に、既存コンポーネントを優先流用する
- Decision: レイアウト・操作配置は design/items_detail.html を再現し、同等 UI 部品が既存ページにあれば再利用する。
- Rationale: デザイン整合と実装効率を両立できる。
- Alternatives considered:
  - 新規コンポーネントを全面作成: 保守コストが増える。
  - 既存画面の微修正のみ: 3点リーダー操作や非活性履歴ボタンの要件適合が不十分になる。

# 機能仕様: アイテム画像アップロード

**フィーチャーブランチ**: `007-item-image-upload`  
**作成日**: 2026-05-04  
**ステータス**: ドラフト

## Clarifications

### Session 2026-05-04

- Q: ユーザーがアイテムに紐付けられた画像を削除したい場合、どのように処理すべきか？ → A: 画像削除ボタンを提供し、クリックするとAzure Blob Storageから画像を削除。アイテムのImageIdはNULLになり、プレースホルダーが表示される。削除時は確認ダイアログを表示する。
- Q: 詳細ページと一覧ページで、画像をどのサイズで表示し、アスペクト比を変更する場合はどのように処理すべきか？ → A: 詳細ページ：最大600x600px、アスペクト比を保持（中央配置）。一覧ページ：80x80px、アスペクト比を保持（中央配置）。オリジナル解像度は1000x1000に正規化。
- Q: クライアント側で画像をキャッシュし、パフォーマンスを最適化する場合、どのような戦略を採用すべきか？ → A: ブラウザデフォルトのキャッシュ制御を使用。Azure Blob Storage は Cache-Control: max-age=86400 を設定（1日有効）。ユーザーが画像更新操作後は max-age=0 でキャッシュを無効化。
- Q: ユーザーへのエラー通知や成功確認をどのような方法で提供すべきか？ → A: スナックバーで一時的にメッセージ表示（自動消滅 3 秒後）。エラーは赤、成功は緑で色分け。詳細エラーはモーダルダイアログで表示。
- Q: Azure Blob Storage に保存された画像へのアクセスを、どのように管理・制御すべきか？ → A: バックエンドが画像取得 API を提供。クライアント側は `/api/items/{itemId}/image` でアクセス。バックエンド側で認可チェック（ユーザーがアイテムへの表示権限を持つか確認）を実施。Azure Blob Storage は Private アクセスのまま。

## ユーザーシナリオとテスト

### ユーザーストーリー 1 - 画像のアップロード (優先度: P1)

ユーザーがアイテムの登録・編集ページから、そのアイテムの代表画像として1つの画像をアップロードできる。

**この優先度の理由**: アイテムに画像を紐付けることが本機能の基本となる機能であり、他の機能の前提条件。

**独立テスト**: 新規アイテムを作成し、画像をアップロードして登録することで完全にテストでき、アイテムごとに画像が保持される価値を提供する。

**受け入れシナリオ**:

1. **Given** アイテム追加編集ページが開かれた状態で、**When** 画像を選択してアップロードボタンを押すと、**Then** ファイルの検証が実行される
2. **Given** 有効な画像形式（jpg、bmp、png、webp、svg）の10MB以下のファイルが選択された状態で、**When** アップロードボタンを押すと、**Then** Azure Blob Storageに画像が保存され、アイテムに画像が紐付けられる
3. **Given** 無効な形式またはサイズ超過のファイルが選択された状態で、**When** アップロードボタンを押すと、**Then** エラーメッセージが表示され、ユーザーは別のファイルを選択できる

---

### ユーザーストーリー 2 - 詳細ページでの画像表示 (優先度: P1)

ユーザーがアイテム詳細ページを開いたときに、そのアイテムに紐付けられた画像を確認できる。

**この優先度の理由**: アイテムの詳細情報を確認する際に画像が必須の情報であり、アップロード後の確認に必須。

**独立テスト**: 画像をアップロードしたアイテムの詳細ページを開き、画像が表示されることでテスト完了。画像がない場合でもプレースホルダーが表示される。

**受け入れシナリオ**:

1. **Given** 画像が登録されたアイテムの詳細ページを開いた状態で、**When** ページが読み込まれると、**Then** 登録された画像が表示される
2. **Given** 画像が登録されていないアイテムの詳細ページを開いた状態で、**When** ページが読み込まれると、**Then** 「画像が登録されていません」といった代替画像が表示される

---

### ユーザーストーリー 3 - 一覧ページでの画像表示 (優先度: P2)

ユーザーがアイテム一覧ページを閲覧するときに、各アイテムの代表画像をプレビュー表示できる。

**この優先度の理由**: 一覧での視認性を向上させる補助的な機能であり、詳細ページでの確認後の実装。

**独立テスト**: 複数のアイテムを表示し、画像が登録されているアイテムと登録されていないアイテムが混在する一覧をレンダリングすることでテスト完了。

**受け入れシナリオ**:

1. **Given** アイテム一覧ページが表示されている状態で、**When** ページが読み込まれると、**Then** 画像が登録されたアイテムについてはその画像が、登録されていないアイテムについてはプレースホルダーが表示される

---

### ユーザーストーリー 4 - 画像の削除 (優先度: P2)

ユーザーがアイテム詳細ページから、そのアイテムに紐付けられた画像を削除できる。

**この優先度の理由**: 画像の管理（置き換え以上の操作）をサポートする重要機能であり、ユーザーに完全なコントロールを提供。

**独立テスト**: 画像が登録されたアイテムの詳細ページで「画像削除」ボタンをクリックし、Azure Blob Storageから画像が削除され、プレースホルダーが表示されることでテスト完了。

**受け入れシナリオ**:

1. **Given** 画像が登録されたアイテムの詳細ページを開いた状態で、**When** 「画像削除」ボタンをクリックすると、**Then** 削除確認ダイアログが表示される
2. **Given** 削除確認ダイアログで「確認」をクリックした状態で、**When** リクエストが送信されると、**Then** Azure Blob Storageから画像が削除され、アイテムのImageIDがNULLになり、プレースホルダーが表示される
3. **Given** 削除確認ダイアログで「キャンセル」をクリックした状態で、**When** ダイアログが閉じられると、**Then** 画像は削除されず、元の状態を保つ

---

### エッジケース

- 同時に複数の画像をアップロードしようとした場合はどうなるか？ → 1つのみサポートするため、追加操作は無視される
- アップロード中にネットワーク接続が途絶した場合はどうなるか？ → エラーを表示し、ユーザーは再度アップロードを試みることができる
- ファイルのサイズが正確に10MBの場合はアップロード可能か？ → 可能（10MB以下とは10MBを含む）
- 画像の解像度が1000x1000を超える場合はどうなるか？ → 保存前に1000x1000にリサイズされる
- 既存の画像を別の画像に置き換えたい場合はどうなるか？ → 新しい画像をアップロードすると既存の画像が置き換わる
- 削除中にネットワーク接続が途絶した場合はどうなるか？ → エラーを表示し、ユーザーは再度削除を試みることができる。Azure Blob Storageからの削除が成功してもアイテムのImageIDが更新される前に接続が途絶した場合は、管理画面から手動修正が必要
- 複数のユーザーが同時に同じアイテムの画像を削除しようとした場合はどうなるか？ → 最初のリクエストのみ成功し、2番目のリクエストはエラーを表示

## 要件

### 機能要件

- **FR-001**: システムはアイテム追加編集ページから1つの画像ファイルのアップロードを受け付けること
- **FR-002**: システムはアップロードされた画像のファイル形式を検証し、jpg、bmp、png、webp、svgのみを受け付けること
- **FR-003**: システムはアップロードされた画像のファイルサイズを検証し、10MB以下のみを受け付けること
- **FR-004**: システムはアップロードされた画像の解像度を取得し、1000x1000ピクセルを超える場合はサーバー側で1000x1000にリサイズして保存すること
- **FR-005**: システムはアップロードされた画像をAzure Blob Storageに保存すること
- **FR-006**: システムはアイテムに紐付けられた画像をアイテム詳細ページに表示すること
- **FR-007**: システムはアイテムに画像が紐付けられていない場合、詳細ページにプレースホルダー画像を表示すること
- **FR-008**: システムはアイテム一覧ページの各アイテムにその代表画像またはプレースホルダーを表示すること
- **FR-009**: システムは新しい画像のアップロード時に既存の画像を置き換えること
- **FR-010**: システムはアップロード処理中にエラーが発生した場合、ユーザーに分かりやすいエラーメッセージを表示すること
- **FR-011**: システムはアイテム詳細ページに「画像削除」ボタンを表示すること（画像が登録されている場合のみ）
- **FR-012**: システムは削除ボタン押下時に確認ダイアログを表示し、ユーザーが明示的に承認してから削除処理を実行すること
- **FR-013**: システムは画像削除時にAzure Blob Storageから物理的にファイルを削除し、アイテムのImageIDをNULLに設定すること
- **FR-014**: システムは詳細ページに画像を最大600x600pxで表示すること。アスペクト比を保持し、中央配置とする
- **FR-015**: システムは一覧ページに画像を80x80pxで表示すること。アスペクト比を保持し、中央配置とする
- **FR-016**: システムはアップロード時にオリジナル画像を1000x1000に正規化して保存すること（サーバーサイド処理）
- **FR-017**: システムは Azure Blob Storage の画像に Cache-Control: max-age=86400（1日）を設定し、ブラウザキャッシュを有効化すること
- **FR-018**: システムは画像アップロード・削除・置き換え操作後に、キャッシュ無効化ヘッダ（max-age=0）を返却すること
- **FR-019**: システムはアップロード成功時にスナックバーで緑色の成功メッセージを表示し、3秒後に自動消滅すること
- **FR-020**: システムはアップロード失敗時にスナックバーで赤色のエラーメッセージを表示し、3秒後に自動消滅すること。詳細エラーはモーダルダイアログで表示すること
- **FR-021**: システムは削除成功時にスナックバーで緑色の確認メッセージを表示し、3秒後に自動消滅すること
- **FR-022**: システムは削除失敗時にスナックバーで赤色のエラーメッセージを表示し、3秒後に自動消滅すること。詳細エラーはモーダルダイアログで表示すること
- **FR-023**: システムは画像取得 API エンドポイント `/api/items/{itemId}/image` を提供すること
- **FR-024**: システムは画像取得 API 呼び出し時に、リクエストを行ったユーザーがアイテムの表示権限を持つか認可チェックを実施すること
- **FR-025**: システムは認可チェックに失敗した場合、HTTP 403 Forbidden を返却すること
- **FR-026**: システムは Azure Blob Storage のアクセスレベルを Private に設定し、直接 URL からのアクセスを禁止すること
- **FR-027**: システムはアップロード・置き換え操作において、Azure Blob Storage への書き込みをバックエンド API のみが実行できるように制御すること

### 主要エンティティ

- **Item**: アイテムエンティティ、ImageIdプロパティを持ち、画像への参照を管理する
- **Image**: 画像を表すエンティティ、Azure Blob Storage のファイル参照、アップロード日時、ファイル形式などのメタデータを保有

## 成功基準

### 測定可能な成果

- **SC-001**: ユーザーが5分以内にアイテムへの画像登録を完了できる
- **SC-002**: 画像のアップロード・表示の処理が3秒以内に完了する
- **SC-003**: アップロードされた画像の検証エラーが100%正確に検出される（形式、サイズ、解像度）
- **SC-004**: システムが100個のアイテムを含む一覧ページを2秒以内にレンダリングできる
- **SC-005**: ユーザーが初回試行で画像のアップロードに成功する確率が95%以上
- **SC-006**: 詳細ページの画像表示サイズが600x600px以下に制限され、アスペクト比が保持される
- **SC-007**: 一覧ページの画像表示サイズが80x80pxに統一され、クライアント側で中央配置される

## 前提条件

- Azure Blob Storageへの接続が確立されており、ローカル環境ではAzuriteが起動している
- Azure Blob Storageの認証情報は別途設定されているため、本仕様では設定方法はスコープ外とする
- ユーザーはアイテムの追加・編集権限を持っている
- ブラウザはHTML5 File APIをサポートしている
- アップロードされた画像のメタデータ（名前、サイズ、形式など）は自動的に抽出可能
- 既存のアイテム管理システムのデータモデルは拡張可能な状態である
- 画像は1000x1000に正規化された状態で Azure Blob Storage に保存されるため、クライアント側での追加リサイズは不要
- CSS や JavaScript でアスペクト比を保持する中央配置は、object-fit: contain で実装可能と想定
- Azure Blob Storage の Cache-Control ヘッダはサーバー側で設定可能であり、画像の更新・削除時にヘッダを動的に変更可能
- UI フレームワーク（Vue.js / React など）にスナックバーコンポーネント（またはトースト通知）が存在し、自動消滅タイマー機能をサポート
- バックエンド API は既存のユーザー認証・認可機構を持っており、リクエストユーザーの権限を検証可能
- Azure Blob Storage の Private アクセス制御（アクセス キーによる認証）が有効な状態

# Feature Specification: 物品カテゴリー管理

**Feature Branch**: `004-item-category-management`  
**Created**: 2026-04-26  
**Status**: Draft  
**Input**: User description: "登録済み物品カテゴリーの一覧確認・追加・編集・削除を行い、UI操作をBackend/DBへ反映する。UIはdesign/category_list.htmlとdesign/category_dialog.htmlを再現し、既存実装は可能な範囲で流用する。同一名称カテゴリー作成は不可。"

## Clarifications

### Session 2026-04-26

- Q: カテゴリー削除時、そのカテゴリーを参照中の既存アイテムをどう扱いますか？ → A: 削除時に参照アイテムを自動で「未分類」へ付け替える。
- Q: 「未分類」はどのようなカテゴリとして扱いますか？ → A: 「未分類」はシステム予約カテゴリとして常設し、名称変更・削除不可にする。
- Q: アイコン・カラーはどのように選択可能としますか？ → A: アイコン・カラーは定義済み候補からのみ選択可能にする。
- Q: カテゴリー一覧の並び順はどうしますか？ → A: 一覧は常にカテゴリー名の昇順で表示する。

## User Scenarios & Testing *(mandatory)*

## 目的（Why）

- 登録されている物品のカテゴリーを一覧で確認したい。
- 物品のカテゴリーを追加したい。
- 物品のカテゴリーを編集・削除したい。
- 設定画面からカテゴリー管理ページへ迷わず遷移したい。

### User Story 1 - カテゴリー一覧を確認する (Priority: P1)

利用者として、現在登録されているカテゴリーを一覧で把握し、物品登録や分類に必要な候補をすぐ見つけたい。

**Why this priority**: 一覧表示が成立しないと追加・編集・削除の導線価値が成立せず、業務の前提を満たせないため。

**Independent Test**: カテゴリー管理画面を開いた際に、保存済みカテゴリーが表示され、名称・アイコン・カラーを視認できれば独立して価値を提供できる。

**Acceptance Scenarios**:

1. **Given** 利用者が設定画面を表示している, **When** カテゴリー管理メニューを選択する, **Then** カテゴリー一覧画面へ遷移する。
2. **Given** カテゴリーが1件以上登録されている, **When** 利用者がカテゴリ一覧画面を表示する, **Then** すべてのカテゴリーが名称・アイコン・カラー付きで表示される。
3. **Given** カテゴリーが未登録である, **When** 利用者がカテゴリ一覧画面を表示する, **Then** 空状態メッセージと追加導線が表示される。

---

### User Story 2 - カテゴリーを追加する (Priority: P1)

利用者として、新しいカテゴリーを追加し、後続の物品登録で使える分類を増やしたい。

**Why this priority**: 分類を新規作成できないと運用を継続できず、機能提供価値が大きく損なわれるため。

**Independent Test**: 追加ボタンからダイアログを開き、名称・アイコン・カラーを入力して保存後、一覧と永続データに反映されれば独立して検証可能。

**Acceptance Scenarios**:

1. **Given** 利用者がカテゴリ一覧画面を表示している, **When** 追加操作を行う, **Then** category_dialog 相当の入力ダイアログが表示される。
2. **Given** ダイアログで有効な名称・アイコン・カラーが入力されている, **When** 保存する, **Then** 新規カテゴリーが作成され一覧に即時反映される。
3. **Given** 既存と同一名称のカテゴリー名を入力している, **When** 保存する, **Then** 作成は拒否され重複不可メッセージが表示される。

---

### User Story 3 - カテゴリーを編集・削除する (Priority: P2)

利用者として、既存カテゴリーの内容を修正したり不要なカテゴリーを削除したりして、分類体系を最新状態に保ちたい。

**Why this priority**: 追加のみでは運用中の名称変更や整理に追従できないが、一覧・追加よりは優先度が一段低いため。

**Independent Test**: 一覧上の対象カテゴリーから編集・削除を行い、成功時に一覧と永続データに反映され、失敗時に安全に通知されることを確認できる。

**Acceptance Scenarios**:

1. **Given** 既存カテゴリーが存在する, **When** 編集操作を行う, **Then** 現在値が初期表示された category_dialog 相当ダイアログが表示される。
2. **Given** 編集後の値が有効で重複していない, **When** 保存する, **Then** 対象カテゴリーのみ更新される。
3. **Given** 編集後の名称が既存カテゴリー名と重複している, **When** 保存する, **Then** 更新は拒否され重複不可メッセージが表示される。
4. **Given** 対象カテゴリーを削除する, **When** 削除確認に同意する, **Then** 対象カテゴリーが一覧および永続データから削除される。

### エッジケース/失敗時の挙動

- 同一名称チェックは前後空白を除去した値で行い、大文字小文字差分のみの重複も同一として扱う。
- 追加・編集リクエスト送信中に通信失敗した場合、画面はクラッシュせず、失敗メッセージを表示して再試行可能な状態を維持する。
- カテゴリー削除対象がすでに別操作で削除済みだった場合、一覧を再取得して最新状態を表示する。
- カテゴリー削除時に参照中アイテムが存在する場合、参照アイテムは自動で「未分類」に再割り当てしてから削除を完了する。
- 「未分類」カテゴリは常設の予約カテゴリとして扱い、名称変更・削除操作を受け付けない。
- カテゴリーの作成・更新時刻は内部および API 応答では UTC で保持し、UI に表示する場合のみ JST に変換する。
- サーバー側バリデーションエラー時は、該当入力項目とエラー理由をダイアログ内で表示する。
- 設定画面にカテゴリー管理導線が表示できない場合は、設定画面自体は利用可能なままにし、該当導線のみエラー表示または非活性表示にする。

## Requirements *(mandatory)*

### 要求（Must / Should / Could）

- **Must**
  - 設定画面からカテゴリー管理画面へ遷移できる導線を提供する。
  - カテゴリー一覧表示（名称・アイコン・カラー）を提供する。
  - カテゴリー一覧はカテゴリー名の昇順で表示する。
  - カテゴリー追加・編集・削除の UI を提供する。
  - 追加・編集は category_dialog デザインを再現したダイアログで行う。
  - アイコン・カラーは定義済み候補から選択できるようにする。
  - UI 操作結果を Backend 経由で DB に保存する。
  - 同一名称カテゴリーの作成・更新を禁止する。
  - category_list/category_dialog の既存デザイン実装がある場合は流用を優先する。
- **Should**
  - 成功・失敗時の状態変化が一覧上で分かりやすく提示される。
  - 通信失敗時に利用者が同じ入力内容で再実行しやすい。
- **Could**
  - アイコン・カラー候補の視認性を高める補助表示（例: プレビュー強調）を提供する。

### Functional Requirements

- **FR-001**: システムは、カテゴリー管理画面で登録済みカテゴリー一覧（名称・アイコン・カラー）をカテゴリー名の昇順で表示しなければならない。
- **FR-002**: システムは、カテゴリー追加操作時に category_dialog 準拠ダイアログを表示し、名称入力と定義済み候補からのアイコン・カラー選択を受け付けなければならない。
- **FR-003**: システムは、カテゴリー編集操作時に対象データを初期値として category_dialog 準拠ダイアログを表示し、名称入力と定義済み候補からのアイコン・カラー再選択を受け付けなければならない。
- **FR-004**: システムは、カテゴリー削除操作時に確認ステップを提供し、同意後は当該カテゴリー参照中のアイテムを「未分類」へ付け替えてから削除処理を実行しなければならない。
- **FR-005**: システムは、追加・編集・削除操作の結果を Backend と通信して永続化し、成功時は一覧表示へ反映しなければならない。
- **FR-006**: システムは、カテゴリー名の重複（前後空白除去および大文字小文字を無視した比較）を許可してはならない。
- **FR-007**: システムは、通信失敗またはバリデーション失敗時にクラッシュせず、利用者が原因を理解できるエラーメッセージを表示しなければならない。
- **FR-008**: システムは、category_list/category_dialog と同等の画面構成を満たしつつ、既存実装がある場合は再利用可能な UI 部品を優先して利用しなければならない。
- **FR-009**: システムは、他利用者による変更の自動同期を提供してはならない（本機能の対象外）。
- **FR-010**: システムは、設定画面にカテゴリー管理ページへの遷移導線を表示し、利用者の選択でカテゴリー管理ページを表示しなければならない。
- **FR-011**: システムは、削除時の付け替え先として使用する「未分類」カテゴリーを有効な状態で保持しなければならない。
- **FR-012**: システムは、「未分類」カテゴリーをシステム予約カテゴリとして常設し、名称変更および削除を許可してはならない。
- **FR-013**: システムは、カテゴリーの作成・更新時刻を内部および API 応答で UTC (ISO 8601 with Z suffix) として保持し、UI に表示する場合のみ JST へ変換しなければならない。

### Key Entities *(include if feature involves data)*

- **カテゴリー**: 物品分類を表すエンティティ。主な属性は名称、アイコン識別子、カラー識別子、識別子、作成/更新時刻。
- **カテゴリー表示候補**: カテゴリー作成・編集時に選択可能なアイコン候補およびカラー候補の定義済み集合。
- **未分類カテゴリー**: 削除時の再割り当て先としてシステムが利用する常設の予約カテゴリーで、名称変更および削除はできない。
- **カテゴリー入力セッション**: 追加・編集ダイアログでの一時入力状態。名称、選択アイコン、選択カラー、バリデーション結果を保持。
- **カテゴリー一覧表示状態**: 表示中のカテゴリー集合、読み込み状態、エラー状態、選択中カテゴリーを持つ画面状態。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 一般利用者テストで、対象者の95%以上が3分以内に「カテゴリ追加」操作を完了できる。
- **SC-002**: 一般利用者テストで、対象者の95%以上が3分以内に「カテゴリ編集」または「カテゴリ削除」操作を完了できる。
- **SC-003**: 重複名称での保存試行に対し、100%で保存拒否と理由表示が確認できる。
- **SC-004**: 追加・編集・削除の成功操作に対し、100%で一覧表示と保存データの結果一致が確認できる。
- **SC-005**: 通信失敗シナリオ試験で、100%のケースで画面クラッシュが発生せず再試行導線が維持される。
- **SC-006**: 利用者テストで、対象者の95%以上が30秒以内に設定画面からカテゴリー管理ページへ到達できる。
- **SC-007**: カテゴリー削除試験で、参照中アイテムを持つケースの100%で削除完了後に対象アイテムが「未分類」へ再割り当てされる。
- **SC-008**: 一覧表示試験で、100%のケースでカテゴリー名が昇順に表示され、追加・編集後も順序が維持される。
- **SC-009**: カテゴリー API の日時フィールド検証で、100%の応答が UTC (`Z`) 形式を返し、UI に表示される日時は JST としてのみ描画される。

## Assumptions

- カテゴリーは物品管理ドメイン内の単一組織スコープで管理され、権限分岐は本仕様では考慮しない。
- アイコン・カラーの選択候補セットは既存デザインおよび既存実装で定義済みの候補を利用する。
- 削除時に他機能への連鎖更新が必要な場合は、Backend 側の既存整合性ルールに従い、本仕様ではUI通知責務のみを持つ。
- 既存 UI 部品の流用可否は現行コードベースで判定できる前提とする。

## 用語

- **カテゴリー**: 物品を分類するための名称付きラベル。
- **category_list**: カテゴリー一覧画面のデザイン参照元。
- **category_dialog**: カテゴリー追加・編集時に使用するダイアログのデザイン参照元。
- **重複名称**: 前後空白を除去し大文字小文字を無視した結果が既存名と一致する名称。
- **予約カテゴリー**: システム都合で保持される特別なカテゴリーで、通常カテゴリーと異なり操作制約を持つ。
